# ADDITIONAL_QUESTIONS

Ответы на дополнительные вопросы брифа ([[SOURCE_BRIEF]] §8, п.43–52). Ссылаются на фактическую
реализацию в `app/` и решения из [[ADR INDEX]].

## 43. Почему API-ключ нельзя хранить в базе в открытом виде?

Открытый ключ в БД — это секрет-эквивалент пароля. При компрометации дампа БД (SQL-инъекция,
утечка бэкапа, доступ инсайдера) все ключи мгновенно используются злоумышленником. Хранение только
хеша делает дамп бесполезным для аутентификации: по хешу нельзя восстановить исходный ключ.

**✅ Реализовано** (`app/core/security.py`, таблица `api_keys`): ключ формата
`company_live_<key_id>_<secret>` — `key_id` хранится открыто (быстрый O(1) lookup), `secret`
высокоэнтропийный (`token_urlsafe(32)`, ≥128 бит). В БД — только **HMAC-SHA256(secret, pepper)**
(серверный pepper из env: даже дамп БД нельзя проверить офлайн против угаданных ключей). Plaintext
показывается один раз при создании; поддержана ротация и `expires_at` (ADR-004, INV-P8).

## 44. Как защитить API от перебора ключей?

- **Высокая энтропия** ключа (≥128 бит) делает перебор статистически невозможным — это первая линия. ✅
- **Rate limiting** на интеграционном входе — ✅ реализован per-client (`app/core/rate_limit.py`);
  для анонимного перебора добавить лимит per-IP на уровне middleware/шлюза (follow-up).
- **Аудит каждой попытки** (`audit_logs`, INV-P10) — ✅ реализован, в т.ч. отказы 401/403/429/422;
  + алерты на всплеск 401 с одного IP (на проде).
- **Постоянное время сравнения** — ✅ `hmac.compare_digest` (`verify_secret` в `app/core/security.py`).
- На периметре: WAF/Cloudflare, временный бан IP после N неудач, CAPTCHA на подозрительные паттерны.

## 45. Что будет, если два администратора одновременно обработают один и тот же запрос?

Сейчас обработка идемпотентна по статусу: при входе проверяется `status == "processed"` → 409
`ALREADY_PROCESSED` (INV-P11). Но между «проверкой» и «записью» есть гонка (race window): оба
прочитали `accepted`, оба прошли проверку, оба выполнили sync → двойная обработка.

**Реализовано (2026-06-16).** Обработка идёт в **единой транзакции** с `SELECT … FOR UPDATE` на
строке запроса (на PostgreSQL): внутренние записи сотрудника используют `commit=False` (flush),
поэтому блокировка строки удерживается до финального commit. Два администратора сериализуются:
второй ждёт, затем видит `processed` → 409 `ALREADY_PROCESSED`. На SQLite (один писатель) FOR UPDATE
— no-op, но семантика единой транзакции сохраняется. `failed` остаётся повторяемым; терминален
только `processed`. См. `app/services/processing.py`, ADR-007. Альтернатива — атомарный
`UPDATE … WHERE id=:id AND status='accepted'` — эквивалентна по гарантиям.

## 46. Как бы ты добавил роли пользователей?

**✅ Реализовано** (ADR-009, Q-4): RBAC на JWT.
- Сущность `users` (email, `password_hash` = scrypt, `role`, `is_active`). Роли: admin / operator / viewer.
- Аутентификация: `POST /api/auth/login` → JWT (HS256); bootstrap-admin сидируется на старте.
- Авторизация: зависимость `require_roles(*roles)` (`app/api/deps.py`) — централизованно (INV-X3):
  `/api/admin/*` → admin; записи employees/tickets → admin|operator; чтения → любой аутентифицированный.
- Интеграционный вход остаётся на `X-API-Key` (машина-машина).
- Дальше — per-resource scopes и blacklist токенов (deferred).

## 48. Как бы ты сделал очередь обработки интеграционных запросов?

**✅ Каркас реализован** (ADR-010): абстракция `TaskQueue` — `InlineTaskQueue` (синхронно, default)
и `ArqTaskQueue` (Redis); воркер `app/worker.py` (arq) с ретраями и **DLQ** (dead-letter → аудит).
Включается через `REDIS_URL`. Дизайн:
- Приём (`POST /api/integration/requests`) сохраняет запрос как `accepted` и (в async-режиме) кладёт
  задачу в очередь — приём быстрый и устойчив к всплескам.
- Воркеры выполняют sync, переводят статус в `processed`/`failed`, пишут `processed_at`/`error_message`.
- Идемпотентность через атомарный переход статуса (✅ реализован, №45) + dedup по идемпотентному ключу.
- Ретраи с экспоненциальной задержкой и DLQ для «ядовитых» сообщений; алерты на DLQ.
- Админский `POST .../process` остаётся синхронным (явное действие); очередь — путь авто-обработки.

## 49. Что бы ты вынес в Redis?

Бэкенды за `REDIS_URL` уже есть (✅ `RedisRateLimiter`, `ArqTaskQueue`); по умолчанию — in-process фолбэк.
- **Rate limiting** — общий счётчик per-client/per-IP между инстансами (✅ `RedisRateLimiter`;
  default in-memory не консистентен при горизонтальном масштабировании — `app/core/rate_limit.py`).
- **Брокер очереди** обработки (✅ `ArqTaskQueue`, см. №48).
- **Кэш** «горячих» чтений: справочник сотрудников, агрегаты статистики `/api/tickets/stats`.
- **Идемпотентные locks/ключи** обработки (SETNX) для защиты от гонки (№45).
- Сессии/blacklist токенов при появлении ролей (№46).

## 50. Как бы ты логировал ошибки на production?

- **✅ Структурированные логи** (JSON по `LOG_JSON`) с **request-id** (middleware + контекст,
  `app/core/observability.py`, `logging.py`); уровни INFO для действий (NFR-6), ERROR для исключений.
  Централизованный хендлер — единая точка (INV-X3). Ответ несёт `X-Request-ID`.
- **✅ Sentry-хук** (`SENTRY_DSN`, опционально) для исключений со stack-trace; `INTERNAL_ERROR`
  не отдаёт детали наружу, но полностью логируется внутри.
- Агрегация в ELK/Loki/Datadog, дашборды по кодам ошибок (`VALIDATION_ERROR` vs `INTERNAL_ERROR`),
  алерты на рост 5xx и на всплеск 401 (перебор ключей, №44).
- Никаких секретов в логах (в `BaseApiClient` тестов и в аудите ключи маскируются).

## 51. Как бы ты масштабировал сервис при росте нагрузки?

- **Stateless-приложение** за load balancer → горизонтальное масштабирование инстансов
  (внешнее состояние только в БД/Redis, INV-D*).
- БД: пул соединений, read-replicas для тяжёлых listing/stats, индексы (№47), позже —
  партиционирование `audit_logs`/`integration_requests` по дате.
- Вынести синхронную обработку в очередь и воркеры (№48), масштабировать воркеры отдельно.
- Кэш в Redis (№49), CDN/gateway на периметре, rate limiting на шлюзе.
- Наблюдаемость (метрики латентности/throughput, №50) для поиска узких мест перед масштабированием.

## 52. Какие места в проекте ты считаешь самыми слабыми и почему?

Часть прежних слабых мест уже устранена; ниже — актуальный честный список.

**Устранено:**
- ✅ Гонка при параллельной обработке (№45) — единая транзакция + `FOR UPDATE`.
- ✅ Admin без авторизации — JWT + RBAC (ADR-009).
- ✅ Аудит отклонённых валидацией попыток (пустой request_type) — пишется в аудит.

**Остаётся (по убыванию):**
- **Rate limit / очередь по умолчанию in-process**: корректно при 1 инстансе; для горизонтального
  масштабирования включить Redis/Arq-бэкенды (`REDIS_URL`) — пути есть, но без интеграционной проверки.
- **`external_id` сверх сущности §2.2** (Q-8): модель расширена таблицей `external_identities` —
  согласованное допущение, но это расширение брифа.
- **Поиск `ILIKE %...%`** не использует индекс на больших объёмах — нужен полнотекстовый (GIN/tsvector).
- **Отзыв JWT-токенов** до истечения отсутствует (нужен blacklist в Redis); per-resource scopes не реализованы.
- **`SELECT … FOR UPDATE`** на SQLite — no-op (для dev приемлемо; реальная сериализация — на PostgreSQL).
