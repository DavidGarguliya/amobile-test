# ADDITIONAL_QUESTIONS

Ответы на дополнительные вопросы брифа ([[SOURCE_BRIEF]] §8, п.43–52). Ссылаются на фактическую
реализацию в `app/` и решения из [[ADR INDEX]].

## 43. Почему API-ключ нельзя хранить в базе в открытом виде?

Открытый ключ в БД — это секрет-эквивалент пароля. При компрометации дампа БД (SQL-инъекция,
утечка бэкапа, доступ инсайдера) все ключи мгновенно используются злоумышленником. Хранение только
хеша делает дамп бесполезным для аутентификации: по хешу нельзя восстановить исходный ключ.
Реализация: `app/core/security.py` — генерируем высокоэнтропийный токен (`secrets.token_urlsafe(32)`,
≥128 бит), отдаём plaintext один раз в ответе создания, в БД пишем `sha256` (ADR-004, INV-P8).

## 44. Как защитить API от перебора ключей?

- **Высокая энтропия** ключа (≥128 бит) делает перебор статистически невозможным — это первая линия.
- **Rate limiting** на интеграционном входе (реализован per-client; для анонимного перебора —
  добавить лимит per-IP на уровне middleware/边шлюза).
- **Аудит каждой попытки** (`audit_logs`, INV-P10) + алерты на всплеск 401 с одного IP.
- **Постоянное время сравнения** (хеш-lookup по индексу не зависит от содержимого; при keyed-HMAC
  использовать `hmac.compare_digest`).
- На периметре: WAF/Cloudflare, временный бан IP после N неудач, CAPTCHA на подозрительные паттерны.

## 45. Что будет, если два администратора одновременно обработают один и тот же запрос?

Сейчас обработка идемпотентна по статусу: при входе проверяется `status == "processed"` → 409
`ALREADY_PROCESSED` (INV-P11). Но между «проверкой» и «записью» есть гонка (race window): оба
прочитали `accepted`, оба прошли проверку, оба выполнили sync → двойная обработка.

**Как закрыть:** атомарный условный переход статуса в БД, а не «read-then-write». Варианты:
- `UPDATE integration_requests SET status='processed' WHERE id=:id AND status='accepted'` — кто
  получил `rowcount=1`, тот и обрабатывает; второй получает 0 → 409.
- `SELECT ... FOR UPDATE` (пессимистичная блокировка строки) в транзакции.
- Уникальный guard/идемпотентный ключ обработки.

Это зафиксировано как follow-up в [[IMPLEMENTATION_LEDGER]] и ADR-007.

## 46. Как бы ты добавил роли пользователей?

- Сущности `users` (с хешем пароля) и `roles` (admin, operator, viewer), связь many-to-many.
- Аутентификация админ-поверхности через JWT/сессии (сейчас admin открыт — OPEN QUESTION Q-4).
- Авторизация — зависимость FastAPI `require_role("admin")` на роутерах `/api/admin/*`; проверка
  прав в одном месте (как и обработка ошибок — централизованно, INV-X3).
- RBAC-матрица: operator создаёт/назначает заявки, admin управляет клиентами и обрабатывает
  интеграцию, viewer — только чтение. Расширяемо до per-resource scopes.

## 47. Какие индексы нужно добавить в базу данных?

Уже созданы миграцией `5ed0618a97bd` (под фактические фильтры):
- `employees`: unique(`email`), `department`, `is_active`, `external_id`, `full_name`, `position`
  (последние три — под поиск `search`).
- `tickets`: `status`, `priority`, `employee_id`, `assigned_to`.
- `api_clients`: unique(`api_key_hash`) — критичен для горячего пути аутентификации.
- `integration_requests`: `client_id`, `status`, `request_type`, `created_at`.
- `audit_logs`: `client_id`, `success`, `action`, `created_at`.

Дополнительно при росте: составные индексы под частые комбинации фильтров
(`(client_id, status)`, `(status, priority)`, `(client_id, created_at)`), частичные индексы
(например, `WHERE is_active`), полнотекстовый индекс (GIN/tsvector в PostgreSQL) для поиска по ФИО.

## 48. Как бы ты сделал очередь обработки интеграционных запросов?

- Приём (`POST /api/integration/requests`) только сохраняет запрос как `accepted` и кладёт задачу в
  очередь (Celery/RQ/Arq + Redis или RabbitMQ) — приём становится быстрым и устойчивым к всплескам.
- Воркеры разбирают очередь, выполняют sync, переводят статус в `processed`/`failed`, пишут
  `processed_at`/`error_message`.
- Идемпотентность через атомарный переход статуса (см. №45) + dedup по идемпотентному ключу.
- Ретраи с экспоненциальной задержкой и DLQ (dead-letter) для «ядовитых» сообщений; алерты на DLQ.
- Текущая синхронная обработка остаётся как fallback/админ-операция.

## 49. Что бы ты вынес в Redis?

- **Rate limiting** — общий счётчик per-client/per-IP между инстансами (сейчас in-memory,
  не консистентен при горизонтальном масштабировании — см. `app/core/rate_limit.py`).
- **Брокер очереди** обработки (см. №48).
- **Кэш** «горячих» чтений: справочник сотрудников, агрегаты статистики `/api/tickets/stats`.
- **Идемпотентные locks/ключи** обработки (SETNX) для защиты от гонки (№45).
- Сессии/blacklist токенов при появлении ролей (№46).

## 50. Как бы ты логировал ошибки на production?

- **Структурированные логи** (JSON) с correlation/request-id; уровни INFO для действий (NFR-6,
  реализовано), ERROR для исключений. Централизованный хендлер уже даёт единую точку (INV-X3).
- **Sentry** (или аналог) для исключений со stack-trace, релизами и алертами; `INTERNAL_ERROR`
  не отдаёт детали наружу, но полностью логируется внутри.
- Агрегация в ELK/Loki/Datadog, дашборды по кодам ошибок (`VALIDATION_ERROR` vs `INTERNAL_ERROR`),
  алерты на рост 5xx и на всплеск 401 (перебор ключей, №44).
- Никаких секретов в логах (в `BaseApiClient` тестов и в аудите ключи маскируются).

## 51. Как бы ты масштабировал сервис при росте нагрузки?

- **Stateless-приложение** за load balancer → горизонтальное масштабирование инстансов
  (внешнее состояние только в БД/Redis, INV-D*).
- БД: пул соединений, read-replicas для тяжёлых listing/stats, индексы (№47), позже —
  партиционирование `audit_logs`/`integration_requests` по дате.
- Вынести синхронную обработку в очередь и воркеры (№48), масштабировать воркеры отдельно.
- Кэш в Redis (№49), CDN/gateway на периметре, rate limiting на шлюзе.
- Наблюдаемость (метрики латентности/throughput, №50) для поиска узких мест перед масштабированием.

## 52. Какие места в проекте ты считаешь самыми слабыми и почему?

- **Гонка при параллельной обработке** (№45): проверка статуса не атомарна — главный технический долг.
- **Rate limiting in-memory**: не работает корректно при >1 инстанса; нужен Redis (№49).
- **Admin-поверхность без авторизации** (Q-4): в брифе не описана, оставлена открытой — для прода
  обязательны роли (№46).
- **Синхронная обработка** интеграции: при тяжёлом payload блокирует запрос; нужна очередь (№48).
- **external_id** добавлен как допущение (Q-8) — модель сотрудника расширена сверх брифа, требует
  согласования.
- **Поиск** через `ILIKE %...%` не использует индекс на больших объёмах — нужен полнотекстовый
  индекс (№47).
