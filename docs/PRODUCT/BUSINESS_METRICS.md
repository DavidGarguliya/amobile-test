# BUSINESS_METRICS

Метрики качества, выведенные из критериев оценки брифа ([[SOURCE_BRIEF]] §7) и общих требований.
Это не продуктовые KPI (заказчик их не задаёт), а **критерии приёмки самого тестового задания**,
по которым измеряется готовность. Используются как ориентир приоритизации в [[ROADMAP]].

## Scoring rubric (из брифа, §7)

| Критерий | Баллы | Покрытие в проекте |
|----------|-------|--------------------|
| API сотрудников | 15 | [[REQUIREMENTS]] FR-E*, спеки `test_employees` |
| API заявок | 20 | FR-T*, спеки `test_tickets` |
| Интеграционное API | 25 | FR-I*, спеки `test_integration_*`, `test_audit` |
| Работа с базой данных | 10 | NFR-1, миграции (этап разработки) |
| Архитектура проекта | 10 | [[FINAL_SYSTEM_SPEC]], [[ADR INDEX]] |
| Безопасность | 10 | NFR-7, [[INVARIANTS]] INV-P7..P10, ADR-004 |
| Ошибки и валидация | 5 | NFR-2..NFR-5, конверт ошибок |
| Документация и запуск | 5 | README, [[RUNBOOK]], OpenAPI |
| **Максимум** | **100** | Проходной порог: **70** |

## Производные метрики качества тест-контура

- **Traceability coverage:** доля FR/NFR, покрытых ≥1 позитивным и ≥1 негативным тест-кейсом.
  Цель: 100% FR имеют позитив; все FR с ветками ошибок имеют негатив. Матрица — в [[TEST_PLAN]].
- **Contract coverage:** доля эндпоинтов, для которых проверяется не только статус-код, но и схема
  тела ответа. Цель: 100% эндпоинтов с телом.
- **Error-envelope conformance:** доля негативных кейсов, проверяющих единый конверт ошибки. Цель: 100%.
- **Determinism:** повторный прогон контура на стабильном окружении даёт идентичный результат
  (нет flaky). Цель: 0 flaky.

## Definition of success для этой сессии (bootstrap)

Bootstrap считается успешным, когда: пакет документации собран, тест-контур коллектится и
запускается (тип B → допустимы `red`), CI настроен, всё закоммичено и запушено, основная
разработка не начата.
