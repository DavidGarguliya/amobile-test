## ТЕСТОВОЕ ЗАДАНИЕ ДЛЯ BACKEND-РАЗРАБОТЧИКА
*Три API-модуля: от простого к сложному*

### 1. Общая задача

Необходимо разработать backend-сервис, состоящий из трёх API-модулей разной сложности:

1. API справочника сотрудников.
2. API внутренних IT-заявок.
3. Интеграционное API для внешних систем.

Цель задания — проверить умение кандидата проектировать API, работать с базой данных,
реализовывать бизнес-логику, авторизацию, обработку ошибок и документацию.

---

### 2. ТЗ №1. API справочника сотрудников

#### 2.1. Задача
Разработать API для хранения и управления списком сотрудников организации.

#### 2.2. Сущность «Сотрудник»

| Поле | Описание |
|------|----------|
| id | Уникальный идентификатор |
| full_name | ФИО сотрудника |
| position | Должность |
| department | Отдел |
| phone | Телефон |
| email | Email |
| is_active | Активен / неактивен |
| created_at | Дата создания |
| updated_at | Дата изменения |

#### 2.3. Методы API

**Создание сотрудника**
```
POST /api/employees

{
  "full_name": "Иванов Иван Иванович",
  "position": "Backend Developer",
  "department": "IT",
  "phone": "+79401234567",
  "email": "ivanov@example.com"
}
```

**Получение списка сотрудников**
```
GET /api/employees
```
Должны поддерживаться фильтры: department, is_active, search, page, limit.
```
GET /api/employees?department=IT&is_active=true&page=1&limit=20
```

**Получение сотрудника по ID**
```
GET /api/employees/{id}
```

**Обновление сотрудника**
```
PUT /api/employees/{id}
```

**Деактивация сотрудника**
```
PATCH /api/employees/{id}/deactivate
```
Физически удалять сотрудника из базы нельзя. При деактивации должно меняться поле is_active = false.

#### 2.4. Требования
4. Проверять обязательные поля.
5. Email должен быть уникальным.
6. Должна быть пагинация списка.
7. Должен быть поиск по ФИО, email и должности.
8. Ошибки должны возвращаться в понятном формате.
9. Данные должны храниться в базе данных.

### 3. ТЗ №2. API внутренних IT-заявок

#### 3.1. Задача
Разработать API для создания и обработки внутренних заявок в IT-отдел.

#### 3.2. Сущность «Заявка»

| Поле | Описание |
|------|----------|
| id | Уникальный идентификатор |
| employee_id | ID сотрудника, который создал заявку |
| title | Заголовок заявки |
| description | Описание проблемы |
| priority | Приоритет |
| status | Статус |
| assigned_to | ID сотрудника, которому назначена заявка |
| created_at | Дата создания |
| updated_at | Дата изменения |
| resolved_at | Дата закрытия |

#### 3.3. Значения priority
```
low
medium
high
critical
```

#### 3.4. Значения status
```
new
in_progress
resolved
rejected
```

#### 3.5. Методы API

**Создание заявки**
```
POST /api/tickets

{
  "employee_id": 1,
  "title": "Не работает принтер",
  "description": "Принтер в кабинете 204 не печатает документы",
  "priority": "medium"
}
```
При создании заявка должна получать статус new.

**Получение списка заявок**
```
GET /api/tickets
```
Должны поддерживаться фильтры: status, priority, employee_id, assigned_to, page, limit.
```
GET /api/tickets?status=new&priority=critical&page=1&limit=20
```

**Получение заявки по ID**
```
GET /api/tickets/{id}
```

**Назначение исполнителя**
```
PATCH /api/tickets/{id}/assign

{
  "assigned_to": 2
}
```
После назначения исполнителя статус заявки должен стать in_progress.

**Изменение статуса заявки**
```
PATCH /api/tickets/{id}/status

{
  "status": "resolved"
}
```

#### 3.6. Правила изменения статусов

| Текущий статус | Разрешённые следующие статусы |
|----------------|-------------------------------|
| new | in_progress, rejected |
| in_progress | resolved, rejected |
| resolved | нельзя менять |
| rejected | нельзя менять |

Если заявка переведена в статус resolved, необходимо заполнить поле resolved_at.

**Статистика по заявкам**
```
GET /api/tickets/stats

{
  "total": 120,
  "new": 20,
  "in_progress": 15,
  "resolved": 80,
  "rejected": 5,
  "critical_opened": 3
}
```

#### 3.7. Требования
10. Заявка должна быть связана с сотрудником.
11. Нельзя создать заявку от несуществующего сотрудника.
12. Нельзя назначить заявку на несуществующего сотрудника.
13. Нельзя нарушать правила перехода статусов.
14. Должна быть фильтрация и пагинация.
15. Должна быть статистика по заявкам.
16. Ошибки должны возвращаться в едином формате.

### 4. ТЗ №3. Интеграционное API для внешних систем

#### 4.1. Задача
Разработать защищённое API для приёма данных от внешних систем. API должно принимать запросы
от внешнего клиента, проверять API-ключ, сохранять запрос в базу, записывать аудит и позволять
администратору обработать полученные данные.

#### 4.2. Сущность «API-клиент»

| Поле | Описание |
|------|----------|
| id | Уникальный идентификатор |
| name | Название клиента |
| api_key_hash | Хеш API-ключа |
| is_active | Активен / неактивен |
| requests_limit_per_minute | Лимит запросов в минуту |
| created_at | Дата создания |

API-ключ нельзя хранить в базе в открытом виде. В базе должен храниться только хеш ключа.

#### 4.3. Сущность «Интеграционный запрос»

| Поле | Описание |
|------|----------|
| id | Уникальный идентификатор |
| client_id | ID API-клиента |
| request_type | Тип запроса |
| payload | JSON-данные запроса |
| status | Статус запроса |
| error_message | Текст ошибки |
| created_at | Дата создания |
| processed_at | Дата обработки |

Значения status: accepted, rejected, processed, failed.

#### 4.4. Сущность «Аудит»

| Поле | Описание |
|------|----------|
| id | Уникальный идентификатор |
| client_id | ID API-клиента |
| action | Действие |
| ip_address | IP-адрес |
| user_agent | User-Agent |
| success | Успешно / неуспешно |
| details | Дополнительные данные |
| created_at | Дата создания |

#### 4.5. Методы API

**Создание API-клиента**
```
POST /api/admin/clients

{
  "name": "1C Integration",
  "requests_limit_per_minute": 60
}
```
Ответ:
```
{
  "client_id": 1,
  "api_key": "company_live_xxxxxxxxxxxxx"
}
```
API-ключ должен быть показан только один раз при создании клиента. В базе открытый API-ключ
хранить нельзя.

**Деактивация API-клиента**
```
PATCH /api/admin/clients/{id}/deactivate
```

**Отправка интеграционного запроса**
```
POST /api/integration/requests

X-API-Key: company_live_xxxxxxxxxxxxx

{
  "request_type": "employee_sync",
  "payload": {
    "external_id": "EMP-001",
    "full_name": "Иванов Иван Иванович",
    "department": "IT",
    "position": "Developer",
    "email": "ivanov@example.com"
  }
}
```

#### 4.6. Правила работы интеграционного API
17. Если API-ключ отсутствует — вернуть ошибку 401.
18. Если API-ключ неверный — вернуть ошибку 401.
19. Если клиент деактивирован — вернуть ошибку 403.
20. Если превышен лимит запросов в минуту — вернуть ошибку 429.
21. Если request_type пустой — вернуть ошибку 422.
22. Успешный запрос должен сохраняться в базе.
23. Каждая попытка обращения должна записываться в аудит.
24. В аудит нужно сохранять IP-адрес и User-Agent.

**Получение списка интеграционных запросов**
```
GET /api/admin/integration/requests
```
Должны поддерживаться фильтры: client_id, status, request_type, date_from, date_to, page, limit.
```
GET /api/admin/integration/requests?client_id=1&status=accepted&page=1&limit=20
```

**Обработка интеграционного запроса**
```
POST /api/admin/integration/requests/{id}/process
```
Если request_type = employee_sync, система должна создать или обновить сотрудника в справочнике
сотрудников. Сопоставление сотрудника выполнять по external_id или, если external_id отсутствует, по email.
```
{
  "status": "processed",
  "message": "Employee synchronized successfully"
}
```
Если обработка завершилась ошибкой, статус должен стать failed, а текст ошибки должен быть
записан в поле error_message. Нельзя повторно обрабатывать запрос, если он уже находится в
статусе processed.

**Получение аудита**
```
GET /api/admin/audit
```
Должны поддерживаться фильтры: client_id, success, action, date_from, date_to, page, limit.
```
GET /api/admin/audit?client_id=1&success=true&page=1&limit=20
```

### 5. Общие требования ко всем трём ТЗ

#### 5.1. Технические требования
Кандидат может использовать любой удобный backend-стек: Python/FastAPI, Node.js/NestJS/Express,
Go, Java/Spring Boot, PHP/Laravel или другой согласованный стек.

База данных: PostgreSQL. Допускается SQLite только для упрощённого тестового запуска, но
структура должна позволять перейти на PostgreSQL.

#### 5.2. Требования к проекту
25. README с инструкцией запуска.
26. Описание используемого стека.
27. Структурированный код.
28. Подключение к базе данных.
29. Миграции или SQL-скрипт создания таблиц.
30. Документация API.
31. Swagger/OpenAPI или Postman collection.
32. Валидация входящих данных.
33. Единая обработка ошибок.
34. Логирование основных действий.
35. Минимум несколько тестов на ключевую бизнес-логику.

#### 5.3. Единый формат ошибок
```
{
  "error": true,
  "code": "VALIDATION_ERROR",
  "message": "Поле email обязательно",
  "details": {
    "field": "email"
  }
}
```
Примеры кодов ошибок: VALIDATION_ERROR, NOT_FOUND, UNAUTHORIZED, FORBIDDEN,
RATE_LIMIT_EXCEEDED, INVALID_STATUS_TRANSITION, ALREADY_PROCESSED, INTERNAL_ERROR.
