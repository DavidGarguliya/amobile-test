# TECH_DECISIONS

Краткое описание принятых технических решений (бриф §6, п.41) и список того, что улучшил бы при
реальной разработке (п.42). Полные обоснования — в [[ADR INDEX]].

## Принятые технические решения (п.41)

- **Стек:** Python 3 + FastAPI + SQLAlchemy 2 + Alembic, БД PostgreSQL (SQLite для dev). FastAPI
  даёт OpenAPI/Swagger из коробки (NFR-8), pydantic — валидацию (NFR-4). ADR-002.
- **Слоистая архитектура:** `api (роутеры) → services (бизнес-правила) → repositories (доступ к
  данным) → models (ORM)`. Контроллеры тонкие; инварианты живут в сервисах. [[FINAL_SYSTEM_SPEC]].
- **Единый конверт ошибок** `{error, code, message, details}` через централизованные exception-
  handlers (один источник истины для ошибок, NFR-2/5, INV-X1..X3). ADR-003.
- **Безопасность API-ключей:** генерация высокоэнтропийного токена, хранение только `sha256`-хеша,
  показ plaintext один раз, поиск клиента по индексу хеша. ADR-004, INV-P8.
- **Rate limiting:** per-client, fixed window «в минуту», in-memory (для прода — Redis). ADR-004.
- **Машина состояний заявок:** таблица разрешённых переходов (данные, не разбросанные `if`),
  `resolved`/`rejected` терминальны, `resolved` проставляет `resolved_at`. ADR-005.
- **Soft delete:** деактивация вместо удаления (сотрудники и клиенты). ADR-006, INV-P1.
- **Идемпотентная обработка интеграции:** `processed` нельзя обработать повторно; `employee_sync`
  матчит по `external_id`, затем по `email`. ADR-007.
- **Аудит каждой попытки** интеграции (ip, user_agent, success), в т.ч. неуспешной. INV-P10.
- **Миграции Alembic** автогенерированы из моделей; переносимость SQLite↔PostgreSQL (NFR-1).
- **Тесты:** POM-контур pytest + httpx (ADR-001), 55 кейсов, green.

## Что улучшил бы при реальной разработке (п.42)

1. **Атомарный переход статуса** при обработке интеграции (закрыть гонку двух админов, §45) —
   условный `UPDATE ... WHERE status='accepted'` или `SELECT ... FOR UPDATE`.
2. **Очередь обработки** (Celery/RQ/Arq + Redis/RabbitMQ) вместо синхронной обработки (§48).
3. **Redis** для распределённого rate limit, кэша чтений и идемпотентных локов (§49).
4. **Роли и авторизация** admin-поверхности (RBAC, §46) — сейчас открыта (Q-4).
5. **Наблюдаемость:** структурные JSON-логи с request-id, Sentry, метрики латентности/ошибок (§50).
6. **Полнотекстовый поиск** (GIN/tsvector) вместо `ILIKE %...%` на больших объёмах (§47).
7. **Составные/частичные индексы** под частые комбинации фильтров; партиционирование журналов.
8. **Postman/контрактные тесты** против CI-инстанса (сейчас CI гоняет `--collect-only` + опц. e2e).
9. **Конфигурируемые коды/политики** (201 vs 200, дубликат email) после подтверждения Q-*.
10. **Доменная валидация** телефона/email-нормализации, мягкие лимиты длины полей.
