# ADR-008: Тип тест-контура (B / TDD-spec) и конфигурация окружения

## Status
Accepted

## Context
Готового API не предоставлено — продукт нужно построить ([[REQUIREMENTS]], task type B). `API_BASE_URL`
и способ admin-авторизации в брифе не заданы (OPEN QUESTION Q-4, Q-6). Нужно решить, как контур
ведёт себя без реализации и откуда берёт конфигурацию (INV-X5).

## Decision
- **Тип контура — TDD-спецификация (B).** Контур обязан коллектиться (`pytest --collect-only`) и
  запускаться; против незапущенного/неполного API тесты будут `red`/`error` — это ожидаемо. Статус
  явно коммуницируется в [[STATE]] и отчёте. По мере реализации продукта тесты «зеленеют».
- **Конфигурация только из окружения** (`tests/api/config/settings.py`), без хардкода секретов:
  - `API_BASE_URL` (default `http://localhost:8000`);
  - admin-авторизация конфигурируема: `ADMIN_AUTH_HEADER` / `ADMIN_AUTH_TOKEN` (по умолчанию пусто
    → admin открыт, как «вне scope брифа», Q-4);
  - таймауты, ожидаемые коды create (`EXPECT_CREATED_CODE`, default 201) — настраиваемы (Q-5).
- **Гибкие, но строгие проверки:** где бриф не фиксирует точный код/форму (Q-5, Q-7, форма
  пагинации), тест проверяет инвариант (например, «не 2xx + конверт ошибки» или «ключи пагинации
  присутствуют»), а конкретику делает конфигурируемой. Это снижает ложные `red` из-за неоднозначностей.

## Consequences
### Positive
- Контур стабилен к разумным реализациям; нет хардкода; легко указать другой base URL/среду.
- Чёткое разделение «обязательный инвариант» vs «деталь реализации».
### Negative
- Конфигурируемость ожиданий требует документирования в `tests/README.md` и [[TEST_PLAN]].

## Alternatives considered
### Жёстко зафиксировать все коды/формы
Отклонено: породит ложные провалы там, где бриф молчит; ухудшит сигнал.
### Мокать API внутри тестов
Отклонено: контур должен проверять реальный сервис (черный ящик), а не моки.
