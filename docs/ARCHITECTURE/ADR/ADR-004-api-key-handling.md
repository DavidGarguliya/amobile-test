# ADR-004: Хранение и проверка API-ключей, rate limiting

## Status
Accepted

## Context
Бриф §4.2/§4.5/§4.6 и NFR-7 требуют: API-ключ не хранить в открытом виде (только хеш), показывать
plaintext один раз; авторизовать интеграцию по `X-API-Key`; применять лимит запросов в минуту;
аудировать каждую попытку. Инварианты INV-P8..P10.

## Decision
- **Генерация:** непрозрачный токен с префиксом (`company_live_<random>`), достаточная энтропия
  (≥128 бит). Plaintext возвращается только в ответе `POST /api/admin/clients` (INV-P8).
- **Хранение:** в БД — только `api_key_hash`. Алгоритм: криптографический хеш, устойчивый к
  перебору. Для секрета высокой энтропии достаточно SHA-256 с поиском по индексу хеша; допустим
  переход на keyed-hash (HMAC) или медленный хеш — деталь реализации, не контракта.
- **Проверка:** `X-API-Key` → хеш → поиск активного клиента → проверка `is_active` (403 если нет)
  → проверка rate limit (429 при превышении). Отсутствие/неверный ключ → 401 (INV-P9).
- **Rate limit:** per-client, окно «в минуту» (`[ASSUMPTION] Q-3`: фиксированное окно; sliding
  допустим). В продакшене выносится в Redis (бриф §49) — вне scope.
- **Аудит:** каждая попытка (успех/отказ) фиксируется с ip_address и user_agent (INV-P10).

## Consequences
### Positive
- Компрометация БД не раскрывает ключи; одноразовый показ снижает риск утечки.
- Аудит + rate limit ограничивают перебор (бриф §44).
### Negative
- Потеря ключа клиентом ⇒ только перевыпуск (plaintext невосстановим) — это намеренно.
- In-memory rate limit не консистентен между инстансами — отмечено как улучшение (Redis).

## Alternatives considered
### Хранить ключ обратимо (шифрование)
Отклонено: нарушает «нельзя хранить в открытом виде» по духу; хеш безопаснее.
### bcrypt/argon2 на каждый запрос
Отклонено по умолчанию: дорого на горячем пути при высокоэнтропийном ключе; оставлено как опция.
