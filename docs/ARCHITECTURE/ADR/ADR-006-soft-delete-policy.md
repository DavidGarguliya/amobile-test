# ADR-006: Политика мягкого удаления (деактивация)

## Status
Accepted

## Context
Бриф §2.3 запрещает физическое удаление сотрудника; деактивация = `is_active=false`. По аналогии
§4.5 вводит деактивацию API-клиента. Требования FR-E9, FR-I3; инвариант INV-P1.

## Decision
Удаления как операции нет. Вместо неё — деактивация:
- `PATCH /api/employees/{id}/deactivate` → `is_active=false`; запись остаётся, остаётся доступной
  для чтения по id и присутствует в списках при `is_active=false`/без фильтра (INV-P1).
- `PATCH /api/admin/clients/{id}/deactivate` → `is_active=false`; деактивированный клиент при
  интеграционном входе получает 403 `FORBIDDEN` (связь с ADR-004/INV-P9).
- Фильтр `is_active` в списке сотрудников позволяет выбирать активных/неактивных (FR-E4).

## Consequences
### Positive
- Сохраняется история и ссылочная целостность (заявки ссылаются на сотрудников).
- Деактивация обратима бизнес-решением; данные не теряются.
### Negative
- Уникальность email сохраняется и для неактивных — повторное создание с тем же email запрещено
  (`[ASSUMPTION]`: уникальность глобальна; при необходимости — пересмотр через новый ADR).

## Alternatives considered
### Физическое удаление с архивной таблицей
Отклонено: прямо противоречит §2.3.
