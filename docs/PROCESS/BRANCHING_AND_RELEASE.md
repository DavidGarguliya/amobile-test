# BRANCHING_AND_RELEASE

Каноничная git-политика проекта. Этот файл — источник истины по ветвлению/мержу/откату
(ссылается [[AGENTS]] и корневой CLAUDE.md).

## Ветки
- `main` — защищённая, всегда «зелёная» (INV-O3). Прямые пуши в `main` после bootstrap запрещены;
  изменения — через PR.
- `feat/next-stage-baseline` — локальная интеграционная ветка для последовательных слайсов до
  стабилизации. Каждый слайс делается на короткоживущей ветке от неё.
- Короткоживущие ветки слайсов: `feat/<slice>`, `fix/<...>`, `docs/<...>`, `test/<...>`, `chore/<...>`.

## Поток слайса
1. Создать короткоживущую ветку от `feat/next-stage-baseline`.
2. Реализовать слайс; прогнать гейты (тесты из [[TEST_PLAN]], линт).
3. Открыть PR (см. [[PR_GENERATION]]); пройти ревью.
4. **Squash-merge** в `feat/next-stage-baseline` (локально) или в `main` (на GitHub).
5. Следующий слайс — с новой ветки от обновлённой базы. Не продолжать цепочку на старой ветке.

## Bootstrap-исключение
Первичная инициализация (этот пакет) коммитится атомарными коммитами и пушится в `main` напрямую —
это начальный scaffold, ещё нет истории и PR-процесса. Далее действует PR-поток выше.

## Запреты
- **force-push запрещён** (в т.ч. в `main` и общие ветки).
- Нельзя переписывать опубликованную историю.
- Нельзя мержить с падающими гейтами или ослаблять тесты ради зелёного прогона.

## Откат
- Откат изменения — через `git revert` (новый коммит), не через перезапись истории.
- Релизные точки помечаются тегами `milestone/<name>`.

## Конфликт с remote при push
Если push отклонён (non-fast-forward / remote не пуст): сделать `git fetch origin`, показать
расхождение оператору и согласовать стратегию (merge/rebase локально). Никакого `--force`.
