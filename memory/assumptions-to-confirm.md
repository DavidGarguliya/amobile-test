Бриф оставляет пробелы; приняты безопасные дефолты-`[ASSUMPTION]`, помеченные как OPEN QUESTIONS.

- Q-1: тип id — целочисленный автоинкремент (по примерам брифа).
- Q-2: формат/энтропия API-ключа — непрозрачный токен с префиксом `company_live_`, хеш в БД.
- Q-3: окно rate limit — фиксированное «в минуту» (sliding допустим).
- Q-4: авторизация admin-эндпоинтов не описана — в тестах конфигурируема через env, по умолчанию открыта.
- Q-5: коды успеха create — 201 для create, 200 для прочих; в тестах настраиваемо.
- Q-6: `API_BASE_URL` не задан — берётся из env, default `http://localhost:8000`.
- Q-7: код дубликата email — 409 (или 422); тест проверяет «не 2xx» + конверт ошибки.
- Q-8: хранение `external_id` сотрудника (нет в сущности §2.2) — нужно для матча employee_sync.

**Почему:** молчаливая подмена недостающих данных запрещена (см. [[feedback-stack-choice]] и
ENGINEER_MODE §6); допущения должны быть явными и проверяемыми.

**Как применять:** перед основной разработкой задать эти вопросы заказчику; при ответах —
обновить `docs/PRODUCT/REQUIREMENTS.md`, затронутые ADR и тесты. Детали — в REQUIREMENTS (раздел
ASSUMPTIONS / OPEN QUESTIONS). См. [[project-scope-and-status]].
